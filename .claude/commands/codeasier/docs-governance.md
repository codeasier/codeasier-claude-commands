参数约定：
- 第一个参数可选，用于指定模式：`audit` 或 `fix`
- 其余参数可选，作为本次文档治理的目标范围（文件、目录、或主题关键词）
- 如果未提供模式，默认使用 `audit`
- 如果未提供范围，就以当前仓库的整体文档体系为范围

模式规则：
- `audit`：审计优先。只做检查与结论输出；除非是极小且显然正确的修正，否则默认不直接修改文件。
- `fix`：整改优先。在低风险范围内直接修复文档问题；高风险改动仍必须先征求用户确认。

如果用户提供了 `$ARGUMENTS`，先按上述规则解析模式与范围；如果第一个参数不是 `audit` 或 `fix`，则将全部参数视为 `scope`，并仍按默认 `audit` 模式执行。

你的任务是进行一次**文档治理**：既要检查问题，也要在合适的模式下决定是否直接整改文档；但对于大规模重组、删除文件、重命名文档、或会影响外部链接/用户习惯的动作，必须先汇报方案并等待用户确认。

核心价值与检查目标：

1. **结构检查**
   - 判断 `README.md` 是否只保留“最小必要信息”：项目是什么、如何快速开始、最关键命令/入口、以及指向 `docs/` 的清晰索引。
   - 如果 README 承载了过多细节、长篇设计说明、重复内容或本应放入 `docs/` 的材料，提出并在允许修改的模式下做低风险整理。
   - 检查 `docs/` 是否存在清晰入口（如索引页、按主题分类、按语言分类），让用户能从 README 顺利进入详情文档。

2. **国际化检查（可选但默认应检查）**
   - 判断仓库是否存在多语言文档场景（如 `README.zh-CN.md`、`docs/zh/`、`docs/en/` 等）。
   - 如果存在多语言：
     - 检查语言分类是否清晰、命名是否一致。
     - 检查 README / docs 索引 / 各语言入口之间的跳转是否指向正确语言版本。
     - 检查是否出现“中文页链到英文分类”或“英文入口链到中文正文”等错误引用。
   - 如果不存在多语言，不要为了形式主义强行引入国际化结构，只需说明当前无需处理。

3. **文档引用检查**
   - 检查 README 与 `docs/` 内部链接、相对路径链接、锚点链接、跨语言链接是否有效。
   - 重点识别：
     - 悬空引用（目标文件不存在）
     - 锚点错误或标题变更后未同步
     - 多语言场景下引用到错误语言
     - README 指向了应已迁移但未迁移的旧文档位置
   - 在允许修改的模式下直接修复明显错误链接。

4. **文档一致性检查**
   - 结合当前仓库真实状态检查文档是否一致，包括但不限于：
     - 项目名称、定位、核心能力
     - 安装方式、运行方式、命令示例
     - 目录结构、功能列表、支持范围
   - 以当前代码、配置、CLI 行为、工作流文件、已存在文档事实为准，避免保留过时描述。
   - 不要编造未实现功能，也不要保留已经失效的承诺。

执行原则：

## 阶段 1：获取上下文并判断治理范围

1. 先读取当前仓库中与文档相关的核心文件，通常至少包括：
   - `README.md`
   - `docs/` 目录下的索引类文档（如果存在）
   - 与项目描述强相关的事实来源，例如 `pyproject.toml`、`package.json`、CLI 入口说明、主要配置文件、关键 workflow 文档等
2. 如果用户提供了 `scope`，优先聚焦该范围，但仍要评估其与 README / docs 索引 / 项目描述之间的关系。
3. 先给出一段简短判断：这是“仅需轻量修正”还是“需要结构整改”。
4. 明确说明本次按 `audit` 还是 `fix` 模式执行。

## 阶段 2：做证据驱动的文档审计

至少输出以下四类检查结论：

1. `README.md` 是否过重，是否应下沉内容到 `docs/`
2. 是否存在多语言结构；如果存在，其入口与引用是否正确
3. 是否存在悬空引用、错误锚点、错误语言引用
4. 文档描述与仓库当前真实状态是否一致

要求：
- 结论必须基于实际文件内容，不要凭空假设。
- 发现问题时，尽量指出具体文件与位置。
- 若某项“看起来可能有问题”但证据不足，要明确标注为待确认，而不是直接定性。

## 阶段 3：根据模式决定是否直接整改

### `audit` 模式
- 默认不修改文件。
- 输出审计结论、整改建议、建议修改顺序。
- 只有在用户随后明确要求整改时，才继续修改。

### `fix` 模式
- 低风险整改直接做，高风险整改先确认。

可以直接做的低风险改动：
- 修正文档中的错误链接、错误锚点、错误语言引用
- 调整 README 中明显冗余或重复的段落，把其压缩为最小必要说明并补充 `docs/` 入口
- 补充或修正 `docs/` 索引中的链接
- 修正与当前项目状态明显不一致、且可由仓库事实直接验证的描述
- 在不改变对外文件路径的前提下，补齐语言入口提示或索引说明

必须先征求用户确认的高风险改动：
- 大规模重写 README 或 docs
- 删除文档文件
- 重命名文档路径或语言目录
- 新建一整套国际化目录结构
- 任何可能影响外部已引用链接的结构迁移

如果属于高风险整改：
1. 先给出建议结构与原因。
2. 明确哪些文件会移动、重命名、删除或新增。
3. 等用户确认后再修改。

## 阶段 4：输出结果

完成后，给用户返回简洁结果，结构如下：

### 执行模式
- `audit` / `fix`

### 审计结论
- README 结构：通过 / 需调整
- 多语言结构：不适用 / 通过 / 需调整
- 文档引用：通过 / 已修复 / 仍有待确认项
- 项目一致性：通过 / 已修复 / 存在阻塞

### 已修改内容
- 若有修改，列出实际改过的文件及改动目的；若无修改，明确说明“本次为 audit 模式，未直接修改文件”

### 建议但未自动执行的事项
- 只列需要用户确认或超出低风险范围的结构性事项

工作边界：
- 目标是“文档治理”，不是写营销文案。
- 不要脱离仓库真实状态凭空补文档。
- 不要顺手修改与文档无关的源码。
- 默认优先复用现有文档结构，而不是另起炉灶。
- 如果仓库几乎没有 `docs/` 体系，也不要强行过度设计；只给出最小、清晰、可落地的整改方案。
